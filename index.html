CMMS/EAM — Software por módulos (estructura + conexión conceptual)

Esta vista es un mapa tipo software (carpetas por módulo). El navegador no puede conectarse directo a PostgreSQL. Para “conectar la base”, se requiere un backend (API) que consulte PostgreSQL y exponga endpoints REST.

Vista Estructura del “software” por módulos

Cada módulo: domain/, db/, api/, events/, handlers/, services/
Backbone: Asset Registry + OT core (Fase 1)
Resto: Inventario/Costos/Fallas/Producción/Compras/SCADA/Calidad/KPIs/Workflow

1 Inventario

  • Subcarpetas: domain/db/api/events/handlers/services
  • Stock + trazabilidad por OT
  • Criticidad y alertas mínimo

2 Costo de mantenimiento + ROI

  • Subcarpetas: domain/db/api/events/handlers/services
  • Costos directos/indirectos
  • ROI por equipo/periodo

3 Fallas en planta + Downtime

  • Subcarpetas: domain/db/api/events/handlers/services
  • Modo/Causa/Efecto
  • Downtime oficial (impacta KPIs)

4 Mantenimiento ↔ Producción

  • Subcarpetas: domain/db/api/events/handlers/services
  • Ventanas + afectación a capacidad
  • Liberación de equipos (gate calidad)

5 Compras integradas (OT → Requisition → OC)

  • Subcarpetas: domain/db/api/events/handlers/services
  • Flujo automatizado desde OT
  • Recepción → inventario + costos

6 Integración SCADA (MQTT) + reglas predictivas

  • Subcarpetas: domain/db/api/events/handlers/services
  • Ingesta de tags (time-series)
  • Disparadores predictivos/preventivos

7 Calidad + Fiabilidad (FMEA/RCA)

  • Subcarpetas: domain/db/api/events/handlers/services
  • AMFE/FMEA
  • RCA coliderado por Calidad

8 Comunicación (Workflow + Notificaciones + Comentarios)

  • Subcarpetas: domain/db/api/events/handlers/services
  • Workflows por aprobaciones
  • Muro de comentarios por OT/casos

9 Uso compartido multi-área (estándares globales)

  • Subcarpetas: domain/db/api/events/handlers/services
  • Catálogos globales + overrides locales
  • Comparabilidad entre plantas

10 KPIs (MTBF/MTTR/Disponibilidad/OEE/Preventivo)

  • Subcarpetas: domain/db/api/events/handlers/services
  • Motor analítico basado en eventos
  • Panel y cumplimiento preventivo

11 Calidad en liberación + auditorías + desviaciones

  • Subcarpetas: domain/db/api/events/handlers/services
  • Auditorías de proceso
  • Release gate + CAPA por desviaciones
Para ejecutar la parte “Software”, debes implementar un backend que exponga endpoints desde estos módulos. Este archivo solo muestra la estructura.

Fase 1 Backbone + Labor + OT + PM

DDL: CMMS_phase1_asset_labor_pm.sql
DDL base: CMMS_schema.sql
Mapa: cmms_structure.html
El archivo SQL es textual (extensión .sql), no HTM/CSS/TSX/TS. Para “ver como software”, ya generaste este mapa en HTML; luego el backend (API) se encarga de conectar PostgreSQL.

Conexión:

Requisito: para conectarte a PostgreSQL desde el front, necesitas un backend (ej. Node/Express, .NET, Java, etc.) que haga las consultas y publique/consume eventos con RabbitMQ.
